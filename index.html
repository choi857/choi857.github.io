网络基础

TCP 和 UDP 的区别

  1. TCP 是面向连接的协议：

    • 发送数据前要先建立连接。
    • 提供可靠的服务，通过 TCP 连接传输的数据不会丢失、没有重复，并且按顺序到达。
    • 类似于打电话，接通后确认身份才开始通信。

  2. UDP 是无连接的协议：

    • 发送数据前不需要建立连接，没有可靠性。
    • 类似于学校广播，直接进行通信，不保证数据一定到达。
    • 支持一对一、一对多、多对一、多对多的通信方式。

  3. TCP 只支持点对点通信，而 UDP 支持多种通信模式。

  4. TCP 是面向字节流的，而 UDP 是面向报文的：

    • TCP 发送数据时以字节为单位，一个数据包可以拆分成若干组进行发送。
    • UDP 一个报文必须一次性发完。

  5. TCP 首部开销（20 字节）比 UDP 首部开销（8 字节）要大：

    • UDP 的主机不需要维持复杂的连接状态表。

Cookie 和 Session 的区别

  1. Cookie：

    • 存储在用户的浏览器中（会话标识符、兴趣爱好等），每次请求都会发送给服务器。
    • 数据存储在客户端，可以被用户查看和修改，不安全。
    • 每个 Cookie 通常有存储容量限制。
    • 可以设置过期时间，可以是会话 Cookie（关闭浏览器后失效）或持久 Cookie（在指定时间后失效）。

  2. Session：

    • 存储在服务器端，客户端只保存一个标识符。
    • 数据存储在服务器端，用户无法直接访问或修改，安全。
    • 理论上没有大小限制，受限于服务器内存或存储设施的大小。
    • 通常有一个过期时间或者在用户退出登录时被销毁，也可以手动清除。

HTTP 和 HTTPS 的区别

  1. 加密机制：

    • HTTP 不加密数据传输。
    • HTTPS 通过 SSL/TLS 加密传输数据。

  2. 安全性：

    • HTTP 传输的数据容易被窃听和篡改。
    • HTTPS 传输的数据经过加密保护，安全性更高。

  3. 端口：

    • HTTP 默认端口号为 80。
    • HTTPS 默认端口号为 443。

  4. 使用场景：

    • HTTP 适用于不涉及隐私和安全要求不高的数据传输。
    • HTTPS 用于保护敏感数据的传输和保障用户隐私。

  5. 默认端口：

    • HTTP 使用的是 80 端口。
    • HTTPS 使用的是 443 端口。

  6. 连接过程：

    • HTTP 使用 TCP 三次握手建立连接。
    • HTTPS 使用 TCP 三次握手加上 TLS（安全传输层）四次握手。

POST 和 GET 的区别

数据传输方式

  • GET：

    • 通过 URL 提交数据，数据在 URL 中可见，以查询字符串的形式附加在 URL 后面。
    • 例如：http://example.com/resource?key1=value1&key2=value2

  • POST：

    • 通过 HTTP 消息主体（body）传输数据，数据不在 URL 中可见。
    • 更适用于传输敏感数据或大量数据。

数据长度限制

  • GET：

    • 受到 URL 长度限制，不同浏览器和服务器对 URL 长度限制不一致，通常在几千字节左右。

  • POST：

    • 通常可以传输更多数据，不受 URL 长度限制。

安全性

  • GET：

    • 数据在 URL 中可见，不适合传输敏感数据，如密码等，容易被截获或泄露。

  • POST：

    • 数据在 HTTP 消息主体中传输，相对更安全，适合传输敏感信息。

幂等性

  • GET：

    • 通常是幂等的，即对同一 URL 的多次请求应该返回相同的结果。

  • POST：

    • 不一定是幂等的，同一 POST 请求可能会导致不同的结果（例如每次提交一个订单会产生一个新的订单号）。

缓存处理

  • GET：

    • 可以被浏览器缓存，因为 GET 请求是幂等的，通常不会对服务器数据产生影响。

  • POST：

    • 不适合被浏览器缓存，因为 POST 请求可能会对服务器数据进行修改或增加，缓存的数据可能不再有效。

设计模式与原则

合成复用原则

定义：尽量使用组合而不是继承来实现代码复用。尽量使用聚合或组合的方式，而不是使用继承。也就是把需要用到的类作为本类的参数、成员变量、局部变量。

优点：

  • 灵活性更高：组合可以在运行时动态地选择不同的实现，而继承在编译时就已经确定。
  • 避免继承层次过深：过度使用继承可能导致类层次结构过于复杂，难以维护。
  • 代码复用性更好：组合可以将多个对象的功能组合在一起，实现更复杂的功能，而继承只能继承单一父类的功能。

单例设计模式

概念：它确保一个类只有一个实例，并提供一个全局访问点。

懒汉模式和饿汉模式的区别

  • 饿汉模式：
    • 在类加载时就创建单例实例。
    • 线程安全，因为实例在类加载时就已经创建，不存在多线程并发问题。
    • 如果实例一直没有被使用，可能浪费资源。
  • 懒汉模式：
    • 在第一次调用 getInstance() 方法时才创建单例实例。
    • 需要处理线程安全问题。

推荐实现方式：

  • 静态内部类：既能保证线程安全，又能实现延迟加载。
  • 双重检查锁：既能保证线程安全，又能实现延迟加载。

模板设计模式

概念：行为型模式，用于描述对象之间的通信和责任分配。

实现方式：

  • 抽象类有一个模板方法和其他行为方法，模板方法按流程调用各行为方法（抽象或非抽象）。
  • 具体子类重写抽象的行为方法。

分布式锁的基本思路

分布式锁的实现思路很简单，就是让一个线程先进来占位，当其他线程进来时，发现已经有人占位了，就会放弃或者稍后再试。

在 Redis 中，占位通常使用 SETNX 指令。先进来的线程先占位，线程的操作执行完成后，再调用 DEL 指令释放位置。

问题与改进

问题 1：锁未释放

如果业务代码执行过程中抛出异常或挂了，DEL 指令可能没有被调用，导致锁无法释放，后续请求会全部堵塞。

解决方案 1：添加过期时间

给锁添加一个过期时间，确保锁在一定时间后能够自动释放。

问题 2：原子性问题

获取锁和设置过期时间是两个操作，不具备原子性。如果在执行这两个操作之间服务器挂了，锁会被占用且无法及时释放。

解决方案 2：使用 SET 命令的 NX 和 EX 选项

从 Redis 2.8 开始，SETNX 和 EXPIRE 可以通过一个命令一起执行。

解决超时问题

问题 3：业务代码耗时

如果业务代码非常耗时，可能会导致锁提前释放，导致多个线程同时执行。

解决方案 3：避免耗时操作

尽量避免在获取锁之后执行耗时操作。

解决方案 4：使用随机字符串和 Lua 脚本

Redis五种数据类型

发表于 2025-03-16 更新于 2025-03-17

Redis 数据类型与常用命令

更多的命令参考官方文档：
http://www.redis.cn/commands.html#

Redis 中的数据以 key/value 的形式存储，五大数据类型主要指的是 value 的数据类型：

  • String
  • List
  • Hash
  • Set
  • Zset

1. String

String 是 Redis 最简单的一种数据结构。在 Redis 中，所有的 key 都是字符串，但不同的 key 对应的 value 可以有不同的数据结构。我们所说的五种不同的数据类型，主要是指 value 的数据类型不同。

Redis 中的字符串是动态字符串，内部是可以修改的，类似于 Java 中的 StringBuilder，它通过分配冗余空间来减少内存的频繁分配。在 Redis 内部结构中，实际分配的内存通常会大于所需的内存。当字符串小于 1MB 时，扩容操作是在现有空间基础上加倍；每次扩容增加 1MB 空间，最大可达 512MB。

String 命令

  • set：给一个 key 赋值。
  • append：如果 key 已存在，直接在对应的 value 后追加值；否则，创建新的键值对。
  • decr：对 value 执行减 1 操作（前提是 value 是数字）。如果 value 不是数字，会报错；如果 value 不存在，则默认值为 0，并在此基础上减一。
  • decrby：与 decr 类似，但可以自定义步长。
  • get：获取一个 key 的 value。
  • getrange：返回 key 对应 value 的子串，类似于 Java 中的 substring。第二个和第三个参数分别是截取的起始和终止位置，其中 -1 表示最后一个字符，-2 表示倒数第二个字符，依此类推。
  • getset：获取并更新某一个 key。
  • incr：给某一个 key 的 value 自增。
  • incrby：给某一个 key 的 value 自增，同时可以设置步长。
  • incrbyfloat：与 incrby 类似，但步长可以是浮点数。
  • mget 和 mset：批量获取和批量存储。
  • ttl：查看 key 的有效期。
  • setex：在给 key 设置 value 的同时，设置过期时间。
  • psetex：与 setex 类似，但时间单位是毫秒。
  • setnx：默认情况下，set 命令会覆盖已存在的 key，而 setnx 则不会。
  • msetnx：批量设置。
  • setrange：覆盖一个已存在的 key 的 value。
  • strlen：查看字符串长度。

BIT 命令

Redis 中的字符串以二进制方式存储。例如，set k1 a，其中 a 对应的 ASCII 码是 97，97 转换为二进制是 01100001。BIT 相关的命令是对二进制进行操作的：

  • getbit：获取 key 对应 value 在 offset 处的 bit 值。
  • setbit：修改 key 对应 value 在 offset 处的 bit 值。
  • bitcount：统计二进制数据中 1 的个数。

2. List

List 是一个简单的字符串列表，按照插入顺序排序。可以从 List 的头部（LEFT）或尾部（RIGHT）插入元素，也可以从头部或尾部弹出元素。

List 命令

  • lpush：将 value 的值从左到右依次插入表头位置。
  • lrange：返回列表指定区间内的元素。
  • rpush：与 lpush 功能类似，但插入方向是从右到左。
  • rpop：移除并返回列表的尾元素。
  • lpop：移除并返回列表的头元素。
  • lindex：返回列表中下标为 index 的元素。
  • ltrim：对列表进行修剪。
  • blpop：阻塞式的弹出，相当于 lpop 的阻塞版。

3. Set

Set 是 String 类型的无序集合，与 List 不同，Set 中的元素不能重复。

Set 命令

  • sadd：添加元素到一个 key 中。
  • smembers：获取一个 key 下的所有元素。
  • srem：移除指定的元素。
  • sismember：返回某一个成员是否在集合中。
  • scard：返回集合的数量。
  • srandmember：随机返回一个元素。
  • spop：随机返回并且出栈一个元素。
  • smove：把一个元素从一个集合移到另一个集合中。
  • sdiff：返回两个集合的差集。
  • sinter：返回两个集合的交集。
  • sdiffstore：类似于 sdiff，但计算结果会保存在一个新的集合中。
  • sinterstore：类似于 sinter，但将交集保存到一个新的集合中。
  • sunion：求并集。
  • sunionstore：求并集并将结果保存到新的集合中。

4. Hash

Hash 类似于 Java 中的 Map，是一个键值对集合，在 Redis 中可以用来存储对象。在 hash 结构中，key 是一个字符串，value 是一个键值对。

Hash 命令

  • hset：添加值。
  • hget：获取值。
  • hmset：批量设置。
  • hmget：批量获取。
  • hdel：删除一个指定的 field。
  • hsetnx：默认情况下，如果 key 和 field 相同，会覆盖已有的 value，而 hsetnx 则不会。
  • hvals：获取所有的 value。
  • hkeys：获取所有的 key。
  • hgetall：同时获取所有的 key 和 value。
  • hexists：返回 field 是否存在。
  • hincrby：给指定的 value 自增。
  • hincrbyfloat：可以自增一个浮点数。
  • hlen：返回某一个 key 中 value 的数量。
  • hstrlen：返回某一个 key 中的某一个 field 的字符串长度。

5. ZSet

ZSet 和 Set 类似，也是 String 类型元素的集合，但 ZSet 中的每个元素都会关联一个 double 类型的分数。ZSet 中的成员是唯一的，但所关联的分数可以重复。

ZSet 命令

  • zadd：将指定的元素添加到有序集合中。
  • zscore：返回 member 的 score 值。
  • zrange：返回集合中的一组元素。
  • zrevrange：返回一组元素，但按倒序排列。
  • zcard：返回元素个数。
  • zcount：返回 score 在某一个区间内的元素数量。
  • zrangebyscore：按 score 的范围返回元素。
  • zrank：返回元素的排名（从小到大）。
  • zrevrank：返回元素的排名（从大到小）。
  • zincrby：score 自增。
  • zinterstore：给两个集合求交集。
  • zrem：弹出一个元素。
  • zlexcount：计算有序集合中成员数量。
  • zrangebylex：返回指定区间内的成员。

2.1 MDC 的作用

MDC 是一个线程安全的上下文存储工具，它允许你在日志中添加额外的信息，如 traceId。这些信息在同一个线程中是共享的，不会影响其他线程。这使得在分布式系统中追踪请求变得非常方便。
